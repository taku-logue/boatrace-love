# Phase 0 要件定義・検証方針 決定版

作成日: 2026-05-26
更新日: 2026-05-27
ステータス: 完了

## 1. Phase 0の目的

Phase 0では、ボートレース予想分析アプリのMVPにおける予測対象、利用データ、検証方法、スコープ外項目を固定する。

このフェーズの最重要目的は、後続フェーズで判断がぶれないように、モデル開発とバックテストのルールを明文化することである。

特に以下を厳格に固定する。

- 予測対象は各艇の1着確率のみ
- 機械学習モデルにオッズを入れない
- MVPは当日朝時点の展示前モデルのみ
- 検証は単勝のみ
- 全国24場の全レースを除外せず扱う
- ランダム分割を禁止し、時系列分割のみ使う
- MVPは完全な個人利用に限定する

## 2. アプリの位置づけ

このアプリは、ボートレースの結果を保証するツールではない。統計モデルと過去データに基づき、各艇の競技的な1着確率を推定し、単勝オッズと比較して期待値を確認するためのローカル分析アプリである。

表示上の中心概念は以下とする。

- 推定1着確率
- 単勝オッズ
- 期待値
- 特徴量による予測根拠
- 過去バックテスト成績

以下のような表現は使わない。

- 必ず当たる
- 鉄板
- 絶対
- 回収率を保証
- 安全な投資

## 3. 予測対象とモデリング方針

## 3.1 予測対象の完全固定

MVPの予測対象は、各艇の「1着確率」のみとする。

モデルは、1レースに出走する各艇を1行として扱い、その艇が1着になるかどうかを二値分類で予測する。

目的変数:

- 1着: `1`
- 2着以下、失格、転覆、欠場など: `0`

出力:

- 各艇の推定1着確率

後処理:

- 同一レース内の各艇の推定確率を正規化し、合計が1.0になるようにする

初期フェーズでは、以下は一切行わない。

- 2連対確率の予測
- 3連対確率の予測
- 2連単予測
- 2連複予測
- 3連単予測
- 3連複予測
- 着順全体のランキング予測
- 自動買い目生成

## 3.2 オッズの完全分離

機械学習モデルの入力特徴量にオッズを含めることを固く禁止する。

モデルの目的は、市場評価をなぞることではなく、選手、枠、モーター、レース場、過去成績、当日朝時点の情報から、純粋な競技的実力としての1着確率を推定することである。

オッズは以下の用途にのみ使う。

- 単勝オッズの保存
- モデル推定1着確率との比較
- 期待値計算
- 単勝バックテスト

期待値の基本式:

```text
期待値 = モデル推定1着確率 × 単勝オッズ
```

禁止事項:

- オッズを学習特徴量に入れる
- 人気順位を学習特徴量に入れる
- オッズ変化率を学習特徴量に入れる
- 市場確率を学習特徴量に入れる
- 記者予想や印を学習特徴量に入れる

## 3.3 利用時点の限定

MVPの予測モデルは、当日朝時点のデータのみを使用する展示前モデルに限定する。

利用可能時点:

- T1: 当日朝

T1で利用できる主な情報:

- 開催日
- 開催場
- レース番号
- 出走選手
- 枠番
- 級別
- 支部
- 性別
- 年齢
- 体重
- 選手過去成績
- 直近成績
- フライング本数
- モーター番号
- モーター過去成績
- ボート番号
- レース場傾向
- 当日朝時点で確定している欠場情報
- 当日朝時点で取得可能な気象情報

T1モデルで使わない情報:

- 展示タイム
- チルト
- スタート展示
- 展示気配
- 展示後の直前情報
- 締切直前オッズ
- レース結果
- 払戻

## 4. データ処理レギュレーション

## 4.1 全レース対象

全国24場の全レースを対象とする。

以下の特殊条件があるレースも、原則としてデータから除外しない。

- 欠場艇があるレース
- 進入固定レース
- 安定板使用
- 周回短縮
- 荒天
- 強風
- 波高が高いレース
- フライング発生レース
- 出遅れ発生レース
- 失格発生レース
- 転覆発生レース
- 6艇未満のレース
- データ欠損があるレース

これらはDropせず、Booleanフラグやカテゴリ特徴量として付与し、判断をモデルに委ねる。

ただし、データ品質上どうしても学習行として成立しないものは、除外ではなく「無効データ」として記録し、件数と理由をログに残す。

## 4.2 特殊条件フラグ

最低限、以下のフラグを持つ。

| フラグ | 型 | 内容 |
|---|---|---|
| `has_absent_boat` | Boolean | 欠場艇がある |
| `is_fixed_entry` | Boolean | 進入固定レース |
| `has_stabilizer` | Boolean | 安定板使用 |
| `is_shortened_lap` | Boolean | 周回短縮 |
| `is_bad_weather` | Boolean | 荒天条件 |
| `has_strong_wind` | Boolean | 強風条件 |
| `has_high_wave` | Boolean | 波高条件 |
| `has_flying` | Boolean | フライング発生 |
| `has_late_start` | Boolean | 出遅れ発生 |
| `has_disqualification` | Boolean | 失格発生 |
| `has_capsize` | Boolean | 転覆発生 |
| `is_less_than_six_boats` | Boolean | 6艇未満 |
| `has_missing_required_data` | Boolean | 必須データ欠損 |

注意:

レース後にしか判明しない `has_flying`、`has_disqualification` などは、学習時の目的変数整理や分析用フラグとしては保存するが、T1推論時の入力特徴量には使わない。

## 4.3 Rawデータ保存

すべての取得データは、正規化前のRawデータを保存する。

保存対象:

- LZHファイル
- LZH解凍後テキスト
- 公式ページから取得したHTML
- 当日出走表HTML
- オッズHTMLまたは取得JSON
- 展示情報HTML
- パース後CSVまたはParquet

Rawデータには以下のメタデータを付与する。

- 取得元URL
- 取得日時
- 対象日
- レース場
- レース番号
- ファイルハッシュ
- パーサーバージョン
- 取得成功・失敗
- 失敗理由

## 5. バックテストのレギュレーション

## 5.1 時系列分割の徹底

データ分割は、以下の時系列分割に固定する。

```text
Train: 過去3年
Valid: Train直後の1年
Test : Valid直後の直近半年
```

例:

```text
Train: 2022-01-01 から 2024-12-31
Valid: 2025-01-01 から 2025-12-31
Test : 2026-01-01 から 2026-06-30
```

実際には利用可能な最新データ日付を基準に、上記の期間幅を保ったまま切り出す。

禁止事項:

- ランダム分割
- K-Fold交差検証
- Stratified K-Fold
- レース単位を崩した行単位シャッフル
- 同一日の一部をTrain、一部をTestに分ける処理

## 5.2 検証対象は単勝のみ

初期モデルの評価と収益性バックテストは、単勝のみで行う。

単勝以外はMVPでは検証しない。

検証しない券種:

- 複勝
- 2連単
- 2連複
- 3連単
- 3連複
- 拡連複

## 5.3 単勝バックテストの基本ルール

単勝バックテストでは、各レースで期待値が最も高い艇を候補とする。

基本計算:

```text
モデル推定1着確率 = p
単勝オッズ = odds
期待値 = p × odds
```

初期の購入判定候補:

- 期待値が1.05以上
- モデル推定1着確率が一定以上
- 必須データ欠損がない
- 単勝オッズが取得できている

ただし、Phase 0時点では購入条件を最終固定しない。Phase 4以降のバックテスト実装時に、複数条件を比較する。

## 5.4 主要評価指標

確率予測の評価:

- Log Loss
- Brier Score
- Calibration Curve
- Top-1 Accuracy

単勝バックテストの評価:

- 回収率
- 的中率
- 投票レース数
- 平均単勝オッズ
- 期待値帯別回収率
- 月次回収率
- レース場別回収率
- グレード別回収率
- 最大ドローダウン

ベースライン比較:

- 1号艇を常に選ぶ
- 人気1位を常に選ぶ
- 全国勝率が最も高い艇を選ぶ
- モーター2連対率が最も高い艇を選ぶ
- 枠番と場別傾向のみの単純モデル

## 6. MVPでやらないこと

## 6.1 3連単への拡張凍結

単勝でのバックテスト回収率が安定して105%を超えるまで、3連単などの複雑な予測には着手しない。

ここでいう「安定して105%を超える」とは、最低でも以下を満たす状態を指す。

- Test期間で回収率105%以上
- 月次で極端な偏りがない
- 特定の場だけに依存していない
- 投票候補数が少なすぎない
- Valid期間だけでなくTest期間でも再現する

## 6.2 記者予想の排除

記者予想はMVPの特徴量に一切入れない。

理由:

- 予測根拠がブラックボックス化しやすい
- 市場評価や人間の主観をモデルがなぞる可能性がある
- SHAPなどによる説明性が落ちる
- 記者ごとの質や更新タイミングの管理が難しい

記者予想は将来的に比較対象として保存する可能性はあるが、MVPの学習・推論には使わない。

## 6.3 展示情報の推論利用なし

展示情報はMVPの学習・推論パイプラインに通さない。

保存のみ行う候補:

- 展示タイム
- チルト
- スタート展示
- 展示気配
- 直前気象

目的:

- 将来の展示後モデル用にRawデータと正規化データを蓄積する
- MVPではT1展示前モデルの精度検証に集中する

## 6.4 オッズ取得頻度の制限

公式サイトへのアクセス負荷とIPブロックのリスクを下げるため、MVPのオッズ取得は1レースあたり以下の3回に限定する。

- 朝の初期オッズ
- 締切15分前
- 締切5分前

オッズはモデル特徴量には使わず、期待値計算と単勝バックテストにのみ使う。

取得時には以下を守る。

- アクセス間隔を空ける
- リトライ回数を制限する
- 失敗時はログを残してスキップする
- 同じページを短時間に連続取得しない
- robots.txt、利用規約、公式サイトの負荷に配慮する

## 6.5 Web公開の見送り

MVPはローカルネットワーク内のDocker環境で動作する完全な個人利用アプリとする。

初期フェーズでは以下を行わない。

- インターネット公開
- ユーザー登録
- 課金
- 不特定多数への提供
- 外部APIとしての公開
- 自動投票機能

理由:

- 法務面の確認が必要
- スクレイピング負荷が増える
- インフラ運用の負荷が増える
- 予測結果の提供責任が重くなる
- まずはモデルとバックテストの妥当性検証に集中するため

## 10. 法務・運用方針

MVPは完全な個人利用とし、外部公開しない。

Phase 0で確認する事項:

- BOAT RACE公式サイトの利用規約
- データダウンロードページの利用条件
- robots.txt
- スクレイピング頻度
- 取得データの再配布可否
- 商用利用可否

確認結果:

| 確認項目 | 確認日 | 参照先 | 判定 | Phase 0での扱い |
|---|---|---|---|---|
| BOAT RACE公式サイトのサイトポリシー | 2026-05-27 | https://www.boatrace.jp/owpc/pc/extra/policy.html | 確認済み | 著作権、禁止事項、免責事項を前提に、私的利用の範囲を超える複製・頒布・公開を行わない。 |
| 大量アクセスに関する禁止事項 | 2026-05-27 | https://www.boatrace.jp/owpc/pc/extra/policy.html | 確認済み | スクレイピングは低頻度、リトライ制限、Raw保存による再取得抑制を必須にする。 |

| 商用利用可否 | 2026-05-27 | https://www.boatrace.jp/owpc/pc/extra/policy.html | 確認済み | MVPでは商用利用しない。外部提供・課金・公開API化は別途確認が完了するまで行わない。 |

運用上の制限:

- 公式サイトへのアクセス頻度を抑える
- オッズ取得は3スナップショットに限定する
- 失敗時に過剰リトライしない
- Rawデータを保存し、再取得を減らす
- インターネット公開しない

Phase 0時点の結論:

- MVPはローカルDocker環境での個人利用に限定する
- 取得データや加工データを第三者へ再配布しない
- Web公開、商用提供、課金、不特定多数への提供は行わない
- 公式サイトへの取得頻度を抑え、大量アクセスにならない実装を必須にする
- 公式ページの取得処理ではUser-Agent、間隔制御、リトライ制限、失敗ログ、Raw保存を実装方針に含める
- サイトポリシーは変更される可能性があるため、本番公開や商用化を検討する時点で再確認する

## 13. MVP成功条件

MVPの成功条件は、3連単や高度な買い目生成ではない。

まず以下を達成する。

- 全国24場の全レースを安定して取り込める
- T1時点の特徴量だけで1着確率を出せる
- オッズを学習に混ぜずに期待値を計算できる
- Train、Valid、Testを時系列で分離できる
- 単勝バックテストを再現できる
- ベースラインと比較できる
- 予測根拠をSHAPなどで確認できる
- ローカル画面で今日のレースと単勝期待値を確認できる

この状態まで到達して初めて、展示後モデル、複勝・連対系、3連単、記者予想、Web公開などの拡張を検討する。
